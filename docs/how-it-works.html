How It Works

VibeThinker loads a quantized 3.1B parameter model into memory and runs inference on CPU via llama-cpp-python. This page explains the internals: model loading, inference, stability mitigations, eval framework, and performance characteristics.

Backend

The worker loads the GGUF file using llama-cpp-python's Llama class with n_ctx=4096 and n_threads=2. On an i5-6500T (2 cores, 4 threads) this uses approximately 3.3 GB of resident memory. The model stays loaded only for the duration of a single request; the worker process exits after generating the response.

Inference

Inference runs on the CPU backend of llama-cpp-python (ggml CPU backend). Typical throughput is ~2 tokens/second on a 2-core i5-6500T. Responses are structured in a think-tag format:

<think>The user wants a fibonacci generator in Rust.
I'll write a function using an iterative approach
with pattern matching for the base cases.</think>
fn fib(n: u32) -> u32 {
    match n {
        0 => 0,
        1 => 1,
        _ => fib(n - 1) + fib(n - 2),
    }
}

The <think> section contains the model's internal reasoning. Most commands strip this tag before displaying output (use --no-think to suppress thinking, or vibe think to show only the reasoning).

Stability

The ggml CPU backend has a known segfault issue after approximately 12-15 consecutive requests. This is a limitation of the llama.cpp CPU backend and not specific to VibeThinker. Three layers of mitigation are in place:

1. Watchdog Auto-Restart

watchdog.sh polls GET /v1/models every 10 seconds. If the endpoint fails 3 times in a row (~30s detection window), it kills and restarts the server process. Total downtime: ~40 seconds worst case.

2. CLI Retry with Exponential Backoff

When the CLI receives a connection error or 5xx response, it retries with exponential backoff: 5s → 10s → 20s. After three failures it gives up with a clear error message. This masks brief restarts transparently.

3. Systemd Restart

The vibethinker.service systemd unit uses Restart=on-failure. If the server exits unexpectedly and the watchdog hasn't already restarted it, systemd will bring it back. Boot survival is guaranteed.

Eval Framework

The built-in vibe eval command runs a suite of 27 tasks across 8 categories to measure model performance. Four grader types are supported:

  • exact — Exact string match against expected output
  • exact_raw — Searches the raw response (before think-tag stripping) to catch answers embedded in reasoning
  • code — Compiles or interprets generated code and checks output
  • rubric — LLM-graded rubric for open-ended tasks

The exact_raw Fix

The exact_raw grader was introduced because the model sometimes embeds the final answer inside the <think> reasoning block. Standard think-tag stripping would remove the answer. exact_raw searches the entire raw response before stripping, catching answers that would otherwise be missed.

Eval results are generated as HTML reports with per-task pass/fail breakdowns. The current pass rate is 96% across 27 tasks.

Performance

End-to-end request latency depends on max_tokens. Typical ranges on an i5-6500T (2 cores):

  • Short prompts (50-100 output tokens): ~25-50 seconds
  • Medium prompts (100-250 output tokens): ~50-125 seconds
  • Long prompts (250-500 output tokens): ~125-250 seconds

Throughput is bound by the CPU backend's single-core token generation. Future work may explore GPU offloading via llama.cpp's CUDA or Metal backends.
